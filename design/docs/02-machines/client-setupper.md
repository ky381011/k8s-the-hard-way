# クライアントセットアッパー

## 用途
kubernetesクラスターを構築するマシンに対してセットアップを行うマシン
Ansibleのみインストールしクラスタのセットアップに必要なツールはインストールしない

## 要件
<details>
<summary>詳細</summary>

- 構築先のマシンに対して必要最低限のアクセスで行う
  - プライベートネットワーク空間
- 高い再利用可能性と可搬性
  - コンテナによる定義
- 自身の複製を作成することが可能

</details>

## 基本設計

<details>
<summary>詳細</summary>

### コンテナ構成
```
Proxmox LXC コンテナ
├─ OS: Ubuntu 22.04 LTS
├─ インストール: Ansible
├─ Ansible Playbook (管理対象)
│  └─ プライベートネットワーク経由で他ノードを構築
└─ 設定ファイル (ボリュームマウント)
```

### リソース仕様
| 項目 | 仕様 |
|------|------|
| CPU | 1 core |
| メモリ | 1GB |
| ストレージ | 10GB |
| ベースイメージ | Ubuntu 22.04 LTS |

### ランタイム構成

#### インストール内容
- **Ansible**: 最新安定版
- **ランタイム依存**: Python 3, OpenSSH, curl, git
- **Kubernetes ツール**: 不含 (kubectl など)

#### 永続ストレージ構成
| パス | マウント元 | 用途 |
|-----|---------|------|
| `/mnt/playbooks` | ホスト playbook リポジトリ | Ansible プレイブック |
| `/mnt/ssh-keys` | ホスト秘密鍵ディレクトリ | SSH 認証用秘密鍵 |
| `/var/log/ansible` | コンテナ内ストレージ | 実行ログ保存 |

#### 実行環境
```
セットアッパーコンテナ (起動時の状態)
├─ Ansible: 設定済みで即実行可能
├─ SSH Agent: playbooks ディレクトリの鍵を自動ロード
├─ 作業用ユーザー: ansible (uid:1000)
└─ 初期状態: ホスト側で定義した playbook 実行可能
```

### ネットワーク仕様
- **IP アドレス**: 10.0.0.10 (固定割り当て)
- **接続方式**: Proxmox Linux ブリッジ (vmbr0)
- **到達可能性**: プライベートネットワーク内のみ (外部通信なし)
- **アウトバウンド SSH**: 対象ノード群へのポート 22 接続

### 動作仕様

#### 初期化フロー (コンテナ起動時)
```
1. Proxmox LXC コンテナ起動
   ↓
2. マウントポイント確認 (/mnt/playbooks, /mnt/ssh-keys)
   ↓
3. SSH Agent 起動と鍵ロード
   ↓
4. Ansible 設定ファイル読み込み
   ↓
5. 準備完了 (playbook 実行待機)
```

#### 実行パターン
- **直接実行**: `ansible-playbook -i /mnt/playbooks/inventories/hosts.ini /mnt/playbooks/site.yml`
- **ホストからのリモート実行**: SSH 経由で playbook 実行
- **手動メンテナンス**: コンテナにログインして対話的に実行

### 制約事項
- Kubernetes 関連ツール非搭載 (診断は playbook 側で実施)
- 外部インターネット接続不可 (プライベートネットワークのみ)
- 複数 playbook 並行実行は非推奨 (順次実行を想定)

### 再利用・運用仕様
- **テンプレート化**: 基本セットアップ完了後にスナップショット化
- **複製**: 新規構築時は複製した新規コンテナを起動
- **状態管理**: マウント先ホスト側でバージョン管理
- **ログ保持**: `/var/log/ansible` を定期バックアップ

## 詳細設計
<details>
<summary>詳細</summary>

### コンテナイメージ設計

#### ビルド構成
- **ベースイメージ**: Ubuntu 22.04 LTS (最小セット)
- **ビルドステージ**: マルチステージビルド未使用（シンプル構成重視）
- **イメージサイズ**: 目標 ~500MB
- **キャッシュ戦略**: 頻繁に変更される内容（ユーザー作成、ディレクトリ準備）は後ろに配置

#### インストール構成要素
- **Ansible**: pip 経由で最新安定版 2.15.x をインストール
- **ランタイム依存パッケージ**: Python 3、OpenSSH クライアント、curl、git、CA 証明書
- **パッケージ最適化**: apt キャッシュは削除してイメージサイズ最小化

#### ユーザー・パーミッション設定
- **作業用ユーザー**: ansible (uid:1000, gid:1000)
- **ホームディレクトリ**: /home/ansible
- **デフォルトシェル**: /bin/bash
- **特権実行**: 不要（rootユーザーでの実行なし）

#### 初期化メカニズム
- **エントリーポイント**: カスタム初期化スクリプトを呼び出し
- **初期化スクリプト配置**: /usr/local/bin/entrypoint.sh
- **実行権限**: 755 (オーナー実行可能)

### Ansible 実行環境設計

#### 設定ディレクトリ構成
- **システムレベル**: /etc/ansible/ にグローバル設定を配置（ansible.cfg、hosts、roles）
- **ユーザーレベル**: /home/ansible/ にユーザー固有設定を配置
- **キャッシュ領域**: .ansible/ ディレクトリにファクト、一時ファイルを保存

#### 主要な設定項目
- **ホスト鍵確認**: 無効化（初回接続時の対話プロンプト回避）
- **ログ出力先**: /var/log/ansible/ansible.log に JSON 形式で記録
- **タイムアウト**: 接続 10秒、リモート実行 10秒
- **実行ユーザー**: 管理対象ノードの root ユーザー
- **並行実行**: forks=5 で 5 ホスト並行実行
- **ファクトキャッシング**: jsonfile キャッシュで 24時間キャッシュ保持
- **インベントリ**: /mnt/playbooks/inventories/hosts.ini を参照

### SSH 鍵管理設計

#### 鍵ローディングプロセス
1. コンテナ起動時に /mnt/ssh-keys ディレクトリの存在確認
2. SSH Agent デーモン起動（環境変数 SSH_AUTH_SOCK に Agent ソケットパスを設定）
3. マウント先から全鍵ファイル (.pem, id_rsa など) を自動検出
4. 検出された各鍵を ssh-add で Agent に登録
5. ssh-add -l で登録確認後、準備完了シグナル

#### 鍵管理ポリシー
| 項目 | 仕様 |
|------|------|
| 鍵の保存場所 | `/mnt/ssh-keys/` (ホストマウント、読み取り専用) |
| ファイルパーミッション | 0600 (所有者読み取り/書き込み) |
| 鍵の形式 | OpenSSH フォーマット (.pem, id_rsa など) |
| 鍵の暗号化 | AES-256 推奨 |
| パスフレーズ | 無しが標準（必要に応じて SSH_ASKPASS で対応） |
| 最大保持鍵数 | 10個まで |
| 鍵の更新 | ホスト側で管理・バージョン管理 |

#### SSH接続の動作仕様
- **ホスト鍵チェック**: 無効（known_hosts ファイルを使用しない）
- **アイデンティティ検索**: 登録済みの全鍵を試行
- **接続試行**: 3回まで再試行
- **接続タイムアウト**: 5秒

#### セキュリティ考慮事項
- Agent ソケット: パーミッション 700 で保護
- 秘密鍵マウント: 読み取り専用マウント点の使用
- ホスト側管理: 本来の鍵管理はホスト側で実施

### ボリュームマウント管理設計

#### マウントポイント詳細仕様
| パス | RO/RW | パーミッション | ホスト側用途 | 説明 |
|-----|-------|----------------|------------|------|
| `/mnt/playbooks` | RW | 755 | playbook リポジトリ | Ansible playbook と inventories の配置 |
| `/mnt/ssh-keys` | RO | 700 | SSH秘密鍵ディレクトリ | 認証用秘密鍵（読み取り専用）|
| `/var/log/ansible` | RW | 755 | ログ出力先 | playbook 実行ログ保存 |

#### マウント検証メカニズム
コンテナ起動時に以下の検証を実施：
- 各マウントポイントの存在確認（必須マウントは失敗時に起動中止）
- パーミッション自動調整（秘密鍵ディレクトリは 700 に強制）
- playbooks ディレクトリはパーミッション 755 に確保（Ansible 実行ユーザーでの読み取り可能性確認）
- マウント成功時にログ出力

#### マウント失敗時の動作
- 必須マウント（playbooks, ssh-keys）が見つからない場合は起動中止
- エラーメッセージをログに出力
- exit コード 1 で終了

### 初期化スクリプト設計

#### entrypoint.sh の5段階初期化フロー
1. **マウントポイント検証**: /mnt/playbooks, /mnt/ssh-keys の存在確認とパーミッション設定
2. **SSH Agent 初期化**: デーモン起動と SSH_AUTH_SOCK 環境変数の設定
3. **SSH 鍵ローディング**: マウント先から全鍵を自動検出し ssh-add で登録、登録確認
4. **ログディレクトリ初期化**: /var/log/ansible の作成とパーミッション設定
5. **Ansible 設定確認**: ansible --version、ansible-inventory コマンドで動作確認

#### 初期化後の動作
- 引数なし起動時: 対話的シェル (/bin/bash) を起動
- 引数あり起動時: 指定コマンドを実行（例: ansible-playbook）

#### エラーハンドリング
- 初期化段階でのエラーは即座に停止（set -e）
- 各段階のエラーメッセージをログに出力
- SSH 鍵ローディングの失敗は warning として継続（鍵ファイルが見つからない場合）

### ヘルスチェック・監視設計

#### ヘルスチェック項目
| 項目 | 確認内容 | 成功条件 |
|------|---------|--------|
| SSH Agent | Agent の起動と鍵登録状態 | ssh-add -l で鍵一覧表示 |
| Ansible | バージョンとコマンド実行可能性 | ansible --version で 2.15.x 表示 |
| インベントリ | playbook リポジトリのインベントリ解析 | ansible-inventory コマンドが正常結果 |
| マウント | 必須マウントポイントの確認 | mount コマンドで /mnt マウント確認 |
| Python | インタプリタとパッケージの動作 | python3 --version で 3.x 表示 |

#### 監視メトリクス
| メトリクス | 取得方法 | 警告閾値 | 対応 |
|----------|---------|--------|------|
| ディスク使用率 | df -h /var/log/ansible | 80% | ログローテーション実施 |
| ログサイズ | du -sh /var/log/ansible | 1GB | 古いログを圧縮/削除 |
| SSH接続数 | netstat で :22 ポート監視 | 監視用 | 異常接続の検出 |
| メモリ使用率 | free コマンド監視 | 800MB | メモリ不足警告 |

#### 監視周期
- **定期監視**: 5分ごと自動チェック
- **イベント監視**: playbook 実行前後に状態確認
- **ログ確認**: 日次で Ansible 実行ログ確認

### ロギング・デバッグ設計

#### ログ出力構成
| ログ種別 | ファイル | フォーマット | 保持期間 | 説明 |
|---------|---------|-----------|--------|------|
| Ansible実行ログ | `/var/log/ansible/ansible.log` | JSON | 30日 | playbook 実行の詳細トレース |
| SSH接続ログ | `/var/log/ansible/ssh.log` | syslog | 30日 | SSH 接続の成功/失敗記録 |
| コンテナイベント | `/var/log/ansible/container.log` | タイムスタンプ付き | 14日 | 起動/停止/エラーイベント |
| デバッグトレース | `/var/log/ansible/debug.log` | 詳細形式 | オンデマンド | -vvvv 実行時の詳細ログ |

#### ログ保持ポリシー
- **日次ローテーション**: 毎日 00:00 にログファイルをローテーション
- **圧縮**: ローテーション済みファイルは gzip で圧縮
- **削除**: 保持期間経過後は自動削除
- **アーカイブ**: 重要なログはホスト側に定期バックアップ

#### デバッグモード有効化
- **環境変数**: ANSIBLE_DEBUG=1, ANSIBLE_VERBOSITY=4 で詳細ログ有効化
- **コマンドライン**: ansible-playbook に -vvvv フラグで 4段階の詳細表示
- **ファイル**: debug.log に全詳細情報をリアルタイム出力

### セキュリティ設計

#### 権限管理ポリシー
| リソース | 所有者 | グループ | パーミッション | 理由 |
|---------|--------|---------|----------------|------|
| `/mnt/ssh-keys` | ansible | ansible | 700 | 秘密鍵保護（所有者のみ読み取り） |
| `/home/ansible` | ansible | ansible | 755 | 作業ディレクトリ（実行ユーザーアクセス） |
| `/var/log/ansible` | ansible | ansible | 755 | ログ書き込み許可（グループ読み取り可） |
| SSH_AUTH_SOCK | ansible | ansible | 700 | Agent ソケット保護（本人のみ） |
| ansible.cfg | ansible | ansible | 644 | 設定ファイル（グローバル読み取り可） |

#### ユーザー・グループ管理
- **作業ユーザー**: ansible (uid:1000)
- **グループ**: ansible (gid:1000)
- **root アクセス**: 原則なし（必要な場合のみ sudo）
- **パスワード**: 不要（SSH 鍵認証のみ）

#### ネットワークセキュリティ
| 項目 | 設定 | 説明 |
|------|------|------|
| 受信ポート | 22 (SSH) | クラスター内のみアクセス許可 |
| 送信先 | 対象ノード群:22 | Ansible playbook 実行対象への SSH |
| DNS | プライベートネットワーク DNS のみ | 外部DNS通信ブロック |
| 外部通信 | ブロック | インターネット接続不可 |

#### SSH セキュリティ設定項目
| 設定項目 | 値 | 説明 |
|---------|-----|------|
| PermitRootLogin | no | root ログイン禁止 |
| PasswordAuthentication | no | パスワード認証無効 |
| PubkeyAuthentication | yes | 公開鍵認証のみ使用 |
| X11Forwarding | no | X11 フォワーディング無効 |
| PermitTunnel | no | トンネリング禁止 |
| AllowAgentForwarding | yes | SSH Agent フォワーディング許可 |
| AllowTcpForwarding | no | TCP フォワーディング禁止 |

#### 秘密鍵保護
- **マウント方式**: 読み取り専用マウント
- **ホスト側管理**: 秘密鍵は本体ホストで厳格に管理
- **アクセス制限**: コンテナからは agent 経由でのアクセスのみ
- **パーミッション**: 0600 強制（コンテナ起動時に自動設定）

### バックアップ・リカバリー設計

#### バックアップ対象
| 項目 | 場所 | 頻度 | 保持期間 | 重要度 |
|------|------|------|--------|--------|
| ログファイル | `/var/log/ansible/` | 日次 | 90日 | 中 |
| SSH設定 | `/home/ansible/.ssh/` | 手動 | 永続 | 高 |
| Ansible キャッシュ | `/home/ansible/.ansible/` | 必要時 | 7日 | 低 |
| Ansible.cfg | `/etc/ansible/ansible.cfg` | 変更時 | 永続 | 中 |

#### スナップショット戦略
1. **初期セットアップ完了後**: Proxmox LXC スナップショット作成
2. **テンプレート化**: 安定動作確認後にテンプレートとして保存
3. **複製運用**: 新規ノード構築時は複製クローンから起動
4. **マウント設定更新**: クローン起動時にホスト側マウント設定を再設定
5. **初期化実行**: entrypoint.sh 実行で環境初期化

#### リカバリープロセス
- **完全復旧**: 最新スナップショットから復元（鍵と playbook は別途同期）
- **部分復旧**: ログディレクトリのクリア → 再初期化
- **鍵復旧**: ホスト側のバージョン管理から復元
- **ロールバック**: 前世代スナップショットへの即座の復帰

#### リカバリーRPO/RTO
| 項目 | 目標値 | 手段 |
|------|--------|------|
| RPO (Recovery Point Objective) | 1日 | 日次ログバックアップ |
| RTO (Recovery Time Objective) | 15分 | スナップショット復元 + マウント設定更新 |

#### パッチ管理
