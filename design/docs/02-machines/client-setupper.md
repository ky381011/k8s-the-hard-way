# クライアントセットアッパー

## 用途
kubernetesクラスターを構築するマシンに対してセットアップを行うマシン
Ansibleのみインストールしクラスタのセットアップに必要なツールはインストールしない

## 要件
- 構築先のマシンに対して必要最低限のアクセスで行う
  - プライベートネットワーク空間
- 高い再利用可能性と可搬性
  - コンテナによる定義
- 自身の複製を作成することが可能

## 基本設計

### コンテナ構成
```
Proxmox LXC コンテナ
├─ OS: Ubuntu 22.04 LTS
├─ インストール: Ansible
├─ Ansible Playbook (管理対象)
│  └─ プライベートネットワーク経由で他ノードを構築
└─ 設定ファイル (ボリュームマウント)
```

### リソース仕様
| 項目 | 仕様 |
|------|------|
| CPU | 1 core |
| メモリ | 1GB |
| ストレージ | 10GB |
| ベースイメージ | Ubuntu 22.04 LTS |

### ランタイム構成

#### インストール内容
- **Ansible**: 最新安定版
- **ランタイム依存**: Python 3, OpenSSH, curl, git
- **Kubernetes ツール**: 不含 (kubectl など)

#### 永続ストレージ構成
| パス | マウント元 | 用途 |
|-----|---------|------|
| `/mnt/playbooks` | ホスト playbook リポジトリ | Ansible プレイブック |
| `/mnt/ssh-keys` | ホスト秘密鍵ディレクトリ | SSH 認証用秘密鍵 |
| `/var/log/ansible` | コンテナ内ストレージ | 実行ログ保存 |

#### 実行環境
```
セットアッパーコンテナ (起動時の状態)
├─ Ansible: 設定済みで即実行可能
├─ SSH Agent: playbooks ディレクトリの鍵を自動ロード
├─ 作業用ユーザー: ansible (uid:1000)
└─ 初期状態: ホスト側で定義した playbook 実行可能
```

### ネットワーク仕様
- **IP アドレス**: 10.0.0.10 (固定割り当て)
- **接続方式**: Proxmox Linux ブリッジ (vmbr0)
- **到達可能性**: プライベートネットワーク内のみ (外部通信なし)
- **アウトバウンド SSH**: 対象ノード群へのポート 22 接続

### 動作仕様

#### 初期化フロー (コンテナ起動時)
```
1. Proxmox LXC コンテナ起動
   ↓
2. マウントポイント確認 (/mnt/playbooks, /mnt/ssh-keys)
   ↓
3. SSH Agent 起動と鍵ロード
   ↓
4. Ansible 設定ファイル読み込み
   ↓
5. 準備完了 (playbook 実行待機)
```

#### 実行パターン
- **直接実行**: `ansible-playbook -i /mnt/playbooks/inventories/hosts.ini /mnt/playbooks/site.yml`
- **ホストからのリモート実行**: SSH 経由で playbook 実行
- **手動メンテナンス**: コンテナにログインして対話的に実行

### 制約事項
- Kubernetes 関連ツール非搭載 (診断は playbook 側で実施)
- 外部インターネット接続不可 (プライベートネットワークのみ)
- 複数 playbook 並行実行は非推奨 (順次実行を想定)

### 再利用・運用仕様
- **テンプレート化**: 基本セットアップ完了後にスナップショット化
- **複製**: 新規構築時は複製した新規コンテナを起動
- **状態管理**: マウント先ホスト側でバージョン管理
- **ログ保持**: `/var/log/ansible` を定期バックアップ
